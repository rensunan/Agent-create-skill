# Agent Creator

## Overview

本技能提供从零到一创建AI智能体的完整流程引导。通过对话式交互逐步明确需求、
设计架构、选型技术、编写代码，最终完成可本地测试的交付物。

## 核心原则

- **选项式交互**：向用户提问时，将选项以"1. xxx  2. xxx  ..."的形式列出，
  用户只需输入数字即可选择。**每轮选项末尾必须包含"0 = 都不满意，我们详细讨论"**，
  当用户选 0 时，以开放式对话深入了解需求。
- **对话驱动**：每个阶段以对话形式推进，确认后再进入下一步。主动引导但不替用户做决定。
- **按需推荐**：技术选型基于已确认的框架和流程，不堆砌无关选项。
- **渐进明确**：从粗到细，从框架到细节，逐步收敛。
- **安全第一**：密钥、API Key 等敏感信息一律使用环境变量或 `.env` 文件，代码中绝不硬编码。
- **可回退**：任何阶段用户不满意或需求变更，可回退到前面任意阶段重新讨论，不必从头开始。

## 选项式交互规范

每次需要用户做选择时，严格遵循以下格式：

```
【问题标题】
1. 选项A — 简短说明
2. 选项B — 简短说明
3. 选项C — 简短说明
0. 都不满意，我们详细讨论
```

- 选项数量控制在 2-5 个，覆盖最常见场景
- 每个选项后附一行简短说明（不超过 15 字），帮助用户理解差异
- 选 0 后，切换为开放式对话，深入追问用户的具体偏好，然后给出针对性建议
- 用户做出选择后，先确认再进入下一问

---

## 入口判断：复杂度评估

在与用户初步交流后，先判断智能体的复杂度级别，决定走完整流程还是精简流程：

| 级别 | 特征 | 流程 |
|------|------|------|
| **简单** | 单一功能、无外部依赖、本地运行、单轮交互 | **精简流程**（阶段一→五→六，跳过框架设计和技术选型，直接用最简方案） |
| **中等** | 多轮对话、有RAG或工具调用、需Web部署 | **完整流程**（六个阶段全部执行） |
| **复杂** | 多智能体协作、多模态、复杂工作流、生产级 | **完整流程**（六个阶段，每个阶段更深入） |

判断依据：功能数量、外部依赖数量、部署要求、是否多智能体协作。

---

## 工作流程

按以下六个阶段顺序推进，每阶段确认后方可进入下一阶段。用户可随时要求回退到前一阶段。

---

### 阶段一：需求梳理

通过选项式对话搞清楚核心需求。按以下顺序逐一提问，每次只问一个问题，
用户输入数字回答后再问下一个。

**第一问：智能体定位**

```
【这个智能体主要帮谁解决什么问题？】
1. 给自己用的效率工具（命令行/本地GUI）
2. 部署成网页给团队或客户用
3. 嵌入现有系统作为功能模块（SDK/API调用）
4. 还没想清楚，先聊聊看
0. 都不满意，我们详细讨论
```

**第二问：部署形态**（若第一问已明确则跳过）

```
【你希望智能体以什么形式运行？】
1. 命令行脚本 — 终端里跑，最简单
2. 本地桌面GUI — 有窗口界面的本地应用
3. Web聊天页面 — 浏览器打开就能用
4. API服务 — 其他程序通过接口调用
0. 都不满意，我们详细讨论
```

**第三问：交互模式**

```
【智能体的交互方式是？】
1. 一问一答 — 用户提问、智能体回复，不记上下文
2. 多轮对话 — 有记忆，能连续聊
3. 任务执行 — 接收指令，自动完成一系列操作
4. 定时触发 — 按预设时间或条件自动运行
0. 都不满意，我们详细讨论
```

**第四问：外部依赖**

```
【需要对接外部系统吗？（可多选，输入数字用逗号分隔，如 1,3）】
1. 需要调用外部API（如天气、新闻、数据库API）
2. 需要读写本地文件
3. 需要访问数据库
4. 需要搜索互联网
5. 都不需要，纯对话即可
0. 说不清楚，帮我分析一下
```

**第五问：预算与成本**

```
【对运行成本有什么限制？】
1. 尽量免费 — 只用免费API和开源模型
2. 低成本 — 每月几十块钱以内可以接受
3. 有预算但不太敏感 — 效果优先
4. 无限制 — 怎么好怎么来
0. 不太确定，帮我分析
```

**第六问：性能与隐私**

```
【对性能和隐私有什么要求？】
1. 数据必须留在本地，不能上传到外部服务
2. 对响应速度要求高（秒回级别）
3. 以上两者都有要求
4. 没有特殊要求，常规即可
0. 还有其他情况，详细说
```

**第七问：用户技术水平**

```
【目标用户的技术水平是？】
1. 普通用户 — 需要简单直观的界面
2. 技术用户 — 可以接受命令行或配置项
3. 开发者 — 需要API接口或可编程能力
4. 我自己用 — 怎么方便怎么来
0. 不确定，帮我分析
```

**输出**：汇总以上 5-7 个回答，生成需求摘要，请用户确认。

---

### 阶段二：框架设计与流程图

基于确认的需求，设计整体框架：

1. **架构分层**：展示系统分为哪些模块/层（如：输入层→处理层→输出层）。
2. **流程图**：使用 Mermaid 绘制流程总览图，标注关键节点和数据流向。Mermaid 代码需确保语法正确、可渲染。
3. **核心组件说明**：每个模块的职责和输入/输出。

**重要**：若用户对框架不满意，以选项式对话搞清楚具体问题点：

```
【你对哪个部分不满意？】
1. 模块划分不合理
2. 数据流向有问题
3. 缺少某个关键环节
4. 整体太复杂，想简化
5. 整体太简单，想增强
0. 其他问题，详细说
```

然后针对性修改设计，直到用户确认满意为止。

**输出**：架构分层描述 + Mermaid 流程图 + 组件说明，待用户确认。

---

### 阶段三：技术选型

基于已确认的框架和流程，按需推荐技术方案。只推荐与当前设计相关的选项，
不列无关清单。**加载 `references/tech-selection-guide.md` 获取详细对照表。**

按以下顺序逐项讨论（跳过不相关的），每项用选项式提问：

**第一项：Agent 框架**

```
【选择 Agent 框架（根据你的需求推荐如下）】
1. 原生 SDK（OpenAI/Anthropic） — 最轻量，直接调用API
2. LangChain — 生态丰富，工具编排能力强
3. LlamaIndex — 数据检索和RAG场景最佳
4. CrewAI — 多智能体分工协作
5. AutoGen — 多智能体自动对话
0. 都不满意，帮我分析选哪个
```

**第二项：LLM 模型**

```
【选择大语言模型】
1. GPT-4o-mini / Claude Haiku — 成本低、速度快的轻量模型
2. GPT-4o / Claude Sonnet — 推理能力强，适合复杂任务
3. DeepSeek-V3 / Qwen — 国内可用，性价比高
4. 本地模型（Ollama） — 数据不出本地，完全免费
0. 不确定，帮我根据前面的需求推荐
```

**第三项：对话记忆**（如需多轮对话）

```
【对话记忆方案】
1. 滑动窗口 — 只记最近几轮，简单够用
2. 摘要压缩 — 自动压缩历史，适合长对话
3. 向量记忆 — 跨会话检索，适合长期使用
4. 数据库持久化 — 需要用户体系和长期存档
0. 不确定，帮我推荐
```

**第四项：RAG**（如需检索外部知识）

```
【是否需要检索增强生成（RAG）？】
1. 需要 — 智能体要基于知识库回答
2. 不需要 — 纯靠模型自身知识就够了
0. 不确定，帮我分析
```

→ 若选 1，追问向量数据库和 Embedding 模型：

```
【向量数据库选型】
1. Chroma — 零配置，适合原型和小规模
2. FAISS — 高性能本地检索
3. Milvus — 生产级分布式
4. Qdrant — 高性能带过滤
5. Pinecone — 全托管免运维
0. 帮我推荐
```

**第五项：前端**（如需Web部署）

```
【选择前端方案】
1. Gradio — 最快出原型，Python一行启动
2. Streamlit — 数据应用和仪表盘
3. Chainlit — 专为LLM对话设计
4. Next.js — 生产级定制化Web应用
0. 帮我推荐
```

**第六项：部署**（如需Web部署）

```
【选择部署方式】
1. 本地运行（localhost）
2. Docker容器化
3. 云平台托管（Vercel/Railway/自建服务器）
0. 帮我推荐
```

其他技术点（评估策略、安全限流、重排序、数据清洗等）在对话中按需提出，
同样使用选项式格式。

**输出**：技术选型清单及理由，待用户确认。

---

### 阶段四：整体确认

汇总前三个阶段的所有决策，展示最终确认清单：

```
【最终确认清单】
📋 需求：xxx
🏗️ 框架：xxx（附流程图）
🔧 技术栈：xxx

以上都确认无误吗？
1. 确认，开始写代码
2. 某个部分需要修改 → 请说明哪个部分
0. 有疑虑，再讨论一下
```

等待用户选择 1 后，方可进入代码编写阶段。

---

### 阶段五：代码编写

**规范要求**：

1. **环境准备**：
   - Python 项目使用虚拟环境（`python -m venv venv`）
   - 依赖写入 `requirements.txt` 并锁定版本号（`package==x.y.z`）
   - 非敏感配置抽取到 `config.yaml` 或 `config.py`

2. **项目结构清晰**：
   ```
   project/
   ├── .env.example      # 环境变量模板（不含真实值）
   ├── .gitignore        # 排除 .env、venv/、__pycache__ 等
   ├── requirements.txt  # 依赖（含版本号）
   ├── config.yaml       # 非敏感配置（可选）
   ├── src/              # 源代码
   │   ├── main.py       # 入口
   │   ├── agent.py      # 智能体核心逻辑
   │   └── ...
   ├── tests/            # 测试
   │   └── test_smoke.py # 冒烟测试
   └── README.md         # 使用说明
   ```

3. **密钥保护**：
   - 所有密钥、Token、API Key 通过环境变量或 `.env` 文件读取
   - 代码中绝不出现硬编码的密钥
   - 交付 `.env.example`（模板，不含真实值）

4. **代码规范**：
   - 单一职责：每个函数/类只做一件事
   - 错误处理：关键路径有 try-except，给出有意义的错误提示
   - 日志：关键节点用 `logging` 输出，方便调试
   - 注释：复杂逻辑加注释，避免废话注释

5. **README.md**：包含项目简介、功能说明、安装步骤、配置说明、启动方式和测试方法

**输出**：完整项目代码，全部写入文件。

---

### 阶段六：测试与交付

1. **本地测试**：
   - 编写并执行 smoketest（冒烟测试），至少验证：
     - 程序能正常启动不报错
     - 能接收用户输入并返回有效输出
     - 核心流程（如：输入→处理→输出）能跑通
   - 如有错误，修复后重新测试直到通过

2. **交付测试接口**：向用户说明本地如何启动和测试，提供完整的测试命令。

3. **等待用户确认**：

```
【测试接口已就绪】
请按 README.md 中的步骤在本地启动并测试。
测试通过后告诉我，我再继续后续操作。

1. 测试通过，继续下一步
2. 测试有问题 → 描述具体报错或异常行为
0. 需要帮助排查
```

4. **后续操作**（用户选择 1 后才执行）：
   - 部署上线
   - 添加高级功能
   - 优化性能
   - 编写更多测试
